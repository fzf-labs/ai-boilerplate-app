# /admin-dev - Admin 前端开发完整工作流

## 描述
管理后台 admin 开发的端到端工作流程，根据不同场景自动选择合适的开发模式，完成从需求到交付的全流程开发。

## 使用方式
```
/admin-dev <需求描述>
```

## 核心工作流程

### 流程 A：标准 CRUD 页面开发

**适用场景**：为后端已有的 CRUD 接口创建标准管理页面

**步骤 1：需求确认**
- 调用 `admin-codeing` 技能
- 确认模块名称、功能描述
- 确认后端接口路径
- 确认特殊字段类型

**步骤 2：代码生成**
- 创建目录结构
- 生成 API 接口定义
- 生成列表页面
- 生成表单组件
- 配置路由

**步骤 3：测试验证**
- 启动开发服务器
- 测试 CRUD 功能
- 验证权限控制

---

### 流程 B：复杂业务页面开发

**适用场景**：自定义页面、复杂交互、多步骤流程

**步骤 1：需求澄清**
- 想法不清晰：调用 `interview` 技能探索方案
- 需要技术选型：调用 `tech-decision` 技能对比方案
- 想法明确：直接进入下一步

**步骤 2：页面设计**
- 组件拆分设计
- 状态管理设计
- API 接口设计

**步骤 3：实现开发**
- 创建组件结构
- 实现业务逻辑
- API 对接
- 状态管理

**步骤 4：测试验证**
- 功能测试
- 交互测试
- 权限测试

---

### 流程 C：页面优化与重构

**适用场景**：优化现有页面性能、重构代码

**步骤 1：问题识别**
- 分析性能瓶颈
- 识别代码质量问题
- 评估用户体验问题

**步骤 2：方案设计**
- 需要对比方案时：调用 `tech-decision` 技能
- 评估组件优化、状态管理、缓存策略等

**步骤 3：实施优化**
- 组件优化（懒加载、虚拟滚动）
- 状态优化（缓存、防抖节流）
- 代码重构

**步骤 4：验证效果**
- 性能测试
- 回归测试
- 用户体验测试

---

## 典型场景示例

### 场景 1：新增标准 CRUD 页面
```
用户输入：/admin-dev 创建部门管理页面

执行流程：
1. 确认后端接口路径：/admin/v1/sys_dept
2. 生成 API 接口定义
3. 生成列表页面和表单组件
4. 配置路由和权限
5. 测试验证
```

### 场景 2：实现复杂业务页面
```
用户输入：/admin-dev 实现 AI 对话页面，支持多轮对话和角色切换

执行流程：
1. 调用 interview 探索设计方案
2. 组件拆分（对话列表、消息输入、角色选择器）
3. 状态管理设计（Pinia store）
4. WebSocket 对接
5. 实现流式响应
6. 测试验证
```

### 场景 3：API 对接
```
用户输入：/admin-dev 对接用户管理的后端接口

执行流程：
1. 查看后端 proto 定义
2. 创建 TypeScript 类型定义
3. 封装 API 函数
4. 在页面中调用
5. 测试验证
```

### 场景 4：性能优化
```
用户输入：/admin-dev 优化订单列表页面，加载很慢

执行流程：
1. 分析当前代码，识别性能问题
2. 调用 tech-decision 对比优化方案
3. 实施优化（虚拟滚动、分页、缓存）
4. 性能测试验证效果
```

---

## 开发注意事项

1. **遵循项目规范**：使用项目既定的代码风格和组件
2. **TypeScript 优先**：所有代码使用 TypeScript
3. **权限控制**：页面和按钮都要配置权限
4. **国际化支持**：使用 $t() 函数处理文本
5. **错误处理**：统一的错误提示和加载状态
